## Add Case Event

<api-endpoint method="POST" path="/discipline/cases/{case_id}/events" />

Add a new event to a discipline case timeline. Events document the progression of a case through investigation, meetings, interventions, and resolutions.

### Path Parameters

<param-field path="case_id" type="uuid" required>
  Unique identifier of the discipline case
</param-field>

### Request Body

<param-field path="event_type" type="string" required>
  Type of event. Common values:
  - `report` - Initial incident report
  - `investigation` - Investigation activity
  - `meeting` - Meeting with students, parents, or staff
  - `intervention` - Disciplinary intervention or action
  - `follow_up` - Follow-up check or status update
  - `resolution` - Resolution or closure activity
  - `note` - General note or documentation
</param-field>

<param-field path="title" type="string" required>
  Brief title summarizing the event
</param-field>

<param-field path="body" type="string" optional>
  Detailed description of what occurred during this event
</param-field>

<param-field path="event_date" type="datetime" required>
  When this event occurred
</param-field>

### Response

<response-field name="id" type="uuid">
  Unique identifier for the event
</response-field>

<response-field name="school_id" type="uuid">
  ID of the school
</response-field>

<response-field name="discipline_case_id" type="uuid">
  ID of the parent discipline case
</response-field>

<response-field name="event_type" type="string">
  Type of event
</response-field>

<response-field name="title" type="string">
  Event title
</response-field>

<response-field name="body" type="string">
  Event description
</response-field>

<response-field name="authored_by" type="uuid">
  User ID of the person who created this event
</response-field>

<response-field name="event_date" type="datetime">
  When the event occurred
</response-field>

<response-field name="created_at" type="datetime">
  When this record was created
</response-field>

<request-example>
```bash
curl -X POST "https://api.athena-erp.com/discipline/cases/123e4567-e89b-12d3-a456-426614174000/events" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "X-School-ID: your-school-id" \
```

## Event Types Reference

Use these standard event types for consistency:

| Event Type | Description | When to Use |
|------------|-------------|-------------|
| `report` | Initial incident report | When first documenting an incident |
| `investigation` | Investigation activities | Interviews, evidence gathering, fact-finding |
| `meeting` | Meetings with stakeholders | Parent conferences, student meetings, committee meetings |
| `intervention` | Disciplinary actions | Warnings, suspensions, behavioral contracts |
| `follow_up` | Progress checks | Monitoring student behavior, checking compliance |
| `resolution` | Case resolution | Final decision, closure of case |
| `note` | General documentation | Any other relevant documentation |

## Timeline Best Practices

<tip>
**Document comprehensively**: The event timeline serves as the official record for due process. Include all significant interactions, decisions, and actions taken.
</tip>

<warning>
**Maintain chronological accuracy**: Always use the actual date/time when events occurred, not when they were recorded in the system.
</warning>

<info>
Events are automatically authored by the currently authenticated user. The `authored_by` field is set automatically and cannot be overridden.
</info>

## Permissions

Event endpoints require one of the following permissions:
- `write:convivencia` - For creating and managing events
- `write:due_process` - For due process management
- `read:all` - For viewing events (read-only)
- `write:all` - Full access to all operations
